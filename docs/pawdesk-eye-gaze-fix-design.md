# Pawdesk 眼球朝向问题分析与修复设计

## 背景

在 `pawdesk` 中，用户反馈：快速拖动宠物一段时间后停下，鼠标明明位于宠物下方，但宠物眼睛仍然看向上方。

该问题与拖动后的窗口位置更新有关，但经过实际验证，它与“宠物快速拖动时逐渐偏移”的问题并不完全相同：眼球问题已经通过一次独立修复解决，而圆周拖动右漂仍然存在。

## 现象定义

复现现象：
- 用户快速拖动宠物。
- 在停止拖动的瞬间，将鼠标停在宠物身体下半部或脚部附近。
- 实际视觉预期应为瞳孔向下偏移。
- 但修复前的行为是：瞳孔仍偏向上方，或明显高于用户认为的视觉中心。

## 初始实现

修复前，眼球偏移逻辑位于：
- `src/renderer/src/pet/usePointerTracking.ts`

其核心计算方式是：
- `centerX = petX + petWidth / 2`
- `centerY = petY + petHeight / 2`

也就是说，系统把“整个窗口的几何中心”当成了眼球朝向参考点。

## 根因分析

### 根因 1：错误使用窗口几何中心代替视觉眼睛中心

`pawdesk` 的宠物窗口是透明窗口，窗口尺寸大于宠物身体可见区域。

相关布局来自：
- `src/main/windows/pet-window.ts`
- `src/renderer/src/styles.css`

关键几何关系：
- 窗口高度：`272`
- `.pet-scene` 顶部偏移：`top: 92px`
- `.pet-face` 位于 body 内：`top: 42px`
- `.pet-eye` 高度：`28px`

因此，眼睛视觉中心的 Y 坐标并不是 `window height / 2 = 136` 所在的窗口中心语义，而应该接近：

- `92 + 42 + 28 / 2 = 148`

也就是说，修复前的参考点并不对应“眼睛所在的位置”，而是对应“整个透明窗口的中心”。

当鼠标位于宠物身体下方时，系统仍可能判断它处于“窗口中心之上”或“接近窗口中心”，因此瞳孔会错误地向上或没有足够向下。

### 根因 2：视觉语义与几何语义混淆

该问题本质上不是符号方向错误，也不是 clamp 范围错误，而是：

- 使用了错误的几何锚点
- 将“窗口中心”误当成“眼睛参考中心”

这意味着：
- 即使拖动事件本身没错
- 即使 `screenY - centerY` 的符号没反
- 只要 `centerY` 选错，视觉结果就会错

### 放大因素：拖动结束后的状态感知

拖动结束后，用户对“鼠标相对宠物”的感知更依赖可见 body/face，而不是整个透明窗口。

因此一旦参考点取窗口中心，问题在“停止后”会显得更明显：
- 用户看到鼠标在宠物下面
- 代码却仍用透明窗口中心判断方向
- 于是形成明显违和感

## 方案比较

### 方案 A：继续使用窗口中心，仅调节比例或 clamp

做法：
- 保留 `petHeight / 2`
- 调整 `/24` 的除数，或增大/缩小 clamp 范围

问题：
- 这是症状修补，不是根因修复
- 会改变眼球运动幅度，但不会修正“参考中心错位”
- 鼠标在宠物下方却看上方的问题仍可能发生

结论：不采用。

### 方案 B：改为使用视觉眼睛锚点

做法：
- 从 CSS 布局提取宠物场景与眼睛的视觉位置常量
- 保持 `centerX` 仍基于窗口/身体水平中心
- 将 `centerY` 改为 `petY + EYE_ANCHOR_Y`

优点：
- 直接修正根因
- 改动小、局部、容易验证
- 不影响现有 x 方向逻辑
- 不改变 clamp 和符号语义

缺点：
- 当前依赖 CSS 常量与 TS 常量一致
- 如果后续样式大改，常量需要同步维护

结论：采用。

### 方案 C：运行时读取 DOM 几何（如 `getBoundingClientRect`）

做法：
- 在 renderer 中实时测量 face/eye 元素位置
- 用真实元素中心作为 gaze anchor

优点：
- 理论上最准确
- 不依赖手写常量与 CSS 保持同步

缺点：
- 复杂度更高
- 引入 DOM 查询与运行时耦合
- 当前问题并不需要这么重的方案

结论：暂不采用，保留为后续增强方向。

## 最终设计

在 `src/renderer/src/pet/usePointerTracking.ts` 中引入以下视觉常量：

- `PET_SCENE_TOP = 92`
- `PET_FACE_TOP = 42`
- `PET_EYE_SIZE = 28`
- `EYE_ANCHOR_Y = PET_SCENE_TOP + PET_FACE_TOP + PET_EYE_SIZE / 2`

保持：
- `centerX = petX + petWidth / 2`
- `x/y` 的 clamp 逻辑不变
- `y = (screenY - centerY) / 24` 的符号不变

仅将：
- `centerY` 从 `petY + petHeight / 2`
- 改为 `petY + EYE_ANCHOR_Y`

## 为什么这个设计有效

因为它将“眼睛应该看向哪里”的参考点，从透明窗口的几何中心，切换成了用户真正感知到的眼睛视觉中心。

这使得：
- 鼠标在眼睛下方时，`screenY - centerY > 0`
- 瞳孔向下偏移
- 鼠标在眼睛上方时，瞳孔向上偏移

整个逻辑与用户视觉判断重新对齐。

## 修改范围

仅涉及：
- `src/renderer/src/pet/usePointerTracking.ts`

相关参考文件：
- `src/renderer/src/styles.css`
- `src/main/windows/pet-window.ts`

## 验证结果

本轮用户手动验证结果表明：
- 眼球朝向问题已经解决

这说明：
- 视觉眼睛锚点设计有效
- 当前剩余问题已收敛到拖动定位本身，而非 gaze 参考中心

## 已知限制与后续注意事项

1. 当前 `EYE_ANCHOR_Y` 仍是基于 CSS 常量的静态推导。
   - 若后续宠物样式、face top、scene top、eye size 发生变化，需要同步更新该常量。

2. x 方向仍使用窗口水平中心。
   - 对当前问题足够，但如果未来宠物左右结构不对称，可能需要进一步使用 face 的真实水平中心。

3. 该修复并没有解决拖动圆周运动时的残余右漂。
   - 这是另一条独立问题线，需要继续分析主进程 drag anchor、轮询时序、窗口位置反馈与系统窗口管理器行为。

## 结论

眼球问题的根因不是“方向算反了”，而是“参考中心取错了”。

通过把 gaze reference 从窗口中心改为视觉眼睛中心，本次修复以最小改动解决了用户可见问题，并且没有改变现有的 x 方向逻辑、符号语义或 clamp 行为。
